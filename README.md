```bash
docker compose exec -T postgres psql -U app -d appdb < db/seed.sql
```


## Анализ и рефакторинг

### 1. Какие проблемы были в исходной реализации списания?

Исходная реализация имела критические проблемы:

- **Отсутствие транзакций**: проверка баланса и создание записи о списании выполнялись в разных операциях, что создавало race condition
- **Неправильный расчет баланса**: функция `getUserBalance` учитывала только начисления (accrual), игнорируя списания (spend) и просроченные начисления
- **Нет идемпотентности**: повторный запрос с тем же параметрами создавал дубликат списания
- **Нет блокировок**: при параллельных запросах несколько операций могли одновременно пройти проверку баланса и списать больше, чем доступно

### 2. Какие проблемы при одновременных запросах вы устранили и как именно?

Проблемы решены через:

- **Транзакции с уровнем изоляции READ_COMMITTED**: все операции списания выполняются атомарно
- **Pessimistic locking (SELECT FOR UPDATE)**: при чтении транзакций пользователя внутри транзакции используется блокировка, которая не позволяет другим запросам изменять данные до завершения текущей транзакции
- **Уникальный индекс на (user_id, request_id)**: на уровне БД гарантируется невозможность создания дубликатов

Теперь при параллельных запросах:
- Первый запрос блокирует строки пользователя
- Остальные запросы ждут завершения первого
- Каждый последующий запрос видит актуальное состояние баланса

### 3. Какие обязательные правила корректности вы обеспечили на уровне БД и на уровне кода?

**На уровне БД (миграция 003):**
- Уникальный составной индекс `(user_id, request_id)` - предотвращает дубли на уровне базы
- Индекс на `expires_at` - ускоряет поиск просроченных начислений
- Существующие constraints: `type IN ('accrual', 'spend')`, `amount > 0`

**На уровне кода:**
- Транзакционность всех операций изменения данных
- Блокировка строк при чтении баланса (`Transaction.LOCK.UPDATE`)
- Проверка баланса перед списанием с учетом просроченных начислений
- Валидация payload при повторном запросе (возврат 409 при несовпадении)

### 4. Как реализована защита от повторного одинакового запроса (duplicate request)?

Идемпотентность реализована через:

1. **Извлечение requestId**: приоритет у header `Idempotency-Key`, fallback на `body.requestId`
2. **Проверка существующей транзакции**: перед созданием новой записи ищем транзакцию с `(user_id, request_id)`
3. **Валидация payload**: если транзакция найдена, проверяем совпадение параметров (amount, type)
   - Совпадает → возвращаем `{ success: true, duplicated: true }`
   - Не совпадает → возвращаем 409 Conflict
4. **Уникальный индекс**: БД гарантирует невозможность создания дубля даже при race condition

Область идемпотентности: `(user_id, requestId)` - разные пользователи могут использовать одинаковые requestId.

### 5. Как обеспечено, что просроченное начисление (expired accrual) не участвует в балансе?

В функции `getUserBalance`:

```typescript
for (const tx of transactions) {
  if (tx.type === 'accrual') {
    // Учитываем только не просроченные начисления
    if (!tx.expires_at || tx.expires_at > now) {
      balance += tx.amount;
    }
  } else if (tx.type === 'spend') {
    balance -= tx.amount;
  }
}
```

Логика:
- Для каждого начисления проверяем `expires_at`
- Если `expires_at === null` - начисление бессрочное, учитываем
- Если `expires_at > now` - еще не истекло, учитываем
- Если `expires_at <= now` - просрочено, пропускаем

Это гарантирует, что при попытке списать бонусы с просроченных начислений вернется ошибка "Not enough bonus".

### 6. Как реализована надежная обработка очереди (повторные попытки, паузы, защита от дублей)?

**Предсказуемый jobId:**
```typescript
await bonusQueue.add('expireAccruals', {...}, { jobId: 'expire-accruals' });
```
Это предотвращает создание дублирующихся задач в очереди.

**Retry с exponential backoff:**
```typescript
{
  autorun: true,
  concurrency: 1,
  settings: {
    backoffStrategy: (attemptsMade) => {
      return Math.min(1000 * Math.pow(2, attemptsMade - 1), 10000);
    }
  }
}
```
- 1-я попытка: немедленно
- 2-я попытка: через 1 секунду
- 3-я попытка: через 2 секунды
- Максимум: 10 секунд

**Защита от дублей бизнес-эффекта:**
```typescript
const requestId = `expire:${accrual.id}`;
const existingSpend = await BonusTransaction.findOne({
  where: { user_id: accrual.user_id, request_id: requestId }
});
if (existingSpend) return; // Уже обработано
```

Каждое просроченное начисление обрабатывается в отдельной транзакции с уникальным `request_id = "expire:<accrual_id>"`. Повторная обработка не создаст дубль благодаря идемпотентности.

### 7. Какие компромиссы вы приняли в рамках ограничения 4–6 часов?

**Принятые компромиссы:**

1. **Простая стратегия блокировки**: используется блокировка всех транзакций пользователя вместо более гранулярной блокировки только активных начислений. Это проще, но может снизить производительность при большом количестве транзакций.

2. **In-memory расчет баланса**: баланс вычисляется в коде путем итерации по всем транзакциям. Для production лучше использовать агрегирующий SQL-запрос или материализованное представление.

3. **Последовательная обработка в очереди**: `concurrency: 1` - обрабатываем задачи по одной. Можно увеличить параллелизм, но это усложнит логику.

4. **Нет оптимизации для частых повторов**: при повторном запросе мы все равно открываем транзакцию. Можно добавить кеш для недавних requestId.

5. **Минимальная обработка ошибок**: в worker логируем ошибки, но не отправляем уведомления. В production нужен мониторинг и алертинг.

6. **Тесты на той же БД**: используем Postgres для тестов вместо изолированной in-memory БД. Это проще настроить, но тесты зависят от внешнего сервиса.

7. **Нет индекса на type**: для оптимизации запросов по типу транзакций можно добавить индекс на `(user_id, type)`, но для текущего объема данных это избыточно.

Все компромиссы приняты осознанно для соблюдения временных рамок, при этом ключевая функциональность и корректность работы обеспечены.
